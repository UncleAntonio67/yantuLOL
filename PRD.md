# 研途LOL (RedDoc) 虚拟商品自动化发货与版权保护系统

本 PRD 基于以下输入整理:

- `原始需求.txt`

- `研途LOL (RedDoc) 虚拟商品自动化发货与版权保护系统 - 开发规划书.txt`

## 1. 背景与目标

### 1.1 背景

团队在小红书售卖科研资料类虚拟商品。当前痛点:

- 发货慢，且依赖人工处理与私信发送。
- 内容容易被二次倒卖，缺少版权追责与传播成本控制。
- 退款后内容无法追回，难以阻断继续访问。

### 1.2 产品目标

- 将“发货”变成标准化流程: 选商品 + 填买家 ID -> 一键生成可发送内容。
- 以方案 B 为主: 不直接发源文件，改为“在线阅读链接 + 独立密码”。
- 支持售后吊销: 退款后立刻阻断在线访问。
- 形成经营数据闭环: 商品、订单、营收、退款，均可统计对账。

### 1.3 非目标与边界

- 不能实现“绝对防传播”。截图/拍照无法完全禁止，目标是可追责与提高传播成本。
- 小红书订单自动同步不在本期硬性范围（除非另行评估可用接口或可接受的导入方式）。

## 2. 角色与权限

### 2.1 角色

- `super_admin`: 超级管理员，拥有全部权限。
- `normal_admin`: 普通管理员，支持日常发货、售后、查看数据。

### 2.2 权限范围（最小可用集）

- 登录与会话: 所有后台用户。
- 商品管理: 仅 `super_admin` 可新增/编辑/上下架。
- 团队成员管理: 仅 `super_admin` 可新增管理员账号。
- 发货与售后: `super_admin` 与 `normal_admin` 均可操作。
- 数据看板: `super_admin` 与 `normal_admin` 均可查看。

## 3. 用户故事（MVP）

### 3.1 后台运营

- 作为管理员，我可以登录后台，进入发货工作台。
- 作为管理员，我可以选择一个商品，输入小红书买家 ID，系统生成一条“在线阅读链接”和“访问密码”，并提供一键复制的发货文案。
- 作为管理员，我可以选择“邮箱发货”，系统自动把链接与密码发到买家邮箱。
- 作为管理员，我可以查看发货记录，按买家 ID / 订单号 / 商品搜索。
- 作为管理员，我可以对某个订单执行退款吊销，吊销后买家无法继续在线阅读。

### 3.2 超级管理员

- 作为超级管理员，我可以新增/编辑商品，包括上传源 PDF（私有存储）。
- 作为超级管理员，我可以新增团队管理员账号并分配角色。

### 3.3 买家阅读端（方案 B 核心）

- 作为买家，我通过链接打开阅读页，输入订单号与密码后可以在线阅读资料。
- 作为买家，如果订单被退款吊销，我在再次访问时会收到“权限已失效”的提示。

## 4. 业务规则与验收标准

### 4.1 在线阅读与吊销

- 系统必须以订单状态为准控制访问。
- 当订单状态为 `refunded` 时:
  - `viewer/auth` 必须返回 403。
  - 已签发 token 在过期前也必须被服务端二次校验阻断（不能只依赖 token 过期）。

### 4.2 水印规则（动态水印）

- 水印在“买家阅读请求”时由服务端在内存中动态叠加，不生成永久的带水印文件落盘。
- 水印内容至少包含:
  - 买家 ID
  - 订单号
  - 时间戳（可选，建议加入）
- 水印样式:
  - 45 度倾斜
  - 低透明度
  - 平铺全页（满屏水印）

### 4.3 防下载策略（降低误操作与传播成本）

- 阅读端不得使用浏览器默认 PDF iframe 直链方式展示。
- 阅读端必须基于 `pdf.js` 渲染到 `canvas`。
- 前端拦截:
  - 右键菜单
  - 常见保存/打印快捷键（Ctrl+S, Ctrl+P）
- 服务端不提供公开可直接下载的静态 PDF URL。

### 4.4 性能

- 单次阅读请求的水印处理应在可接受范围内完成:
  - MVP 目标: 典型文件大小下 P95 < 2s（可先以 5s 为门槛）
  - 若 PDF 过大，必须支持缓存策略或分页加载策略的技术预留

### 4.5 安全

- 后台接口需要 JWT 鉴权。
- 买家阅读端采用“订单号 + 密码”换取短时 token，token 需具备:
  - 短过期（例如 5-15 分钟）
  - 绑定订单
  - 可在退款后立即失效（服务端实时校验订单状态）
- 密码不能明文存储（存 hash），后台如需展示密码，只能在生成时返回一次，后续展示“重置密码”而不是“查看密码”。

## 5. 数据与报表口径

### 5.1 关键指标

- 今日营业额
- 今日发货单数
- 在售商品数
- 累计退款单数
- 商品维度: 销量、营收、退款率

### 5.2 统计口径建议

- “营业额”默认按 `active` 订单累加，退款订单从营收中扣除。
- 时间口径按 `Order.created_at`（发货生成时间）统计。

## 6. 约束与开放问题

范围决策（已确认，2026-03-12）:

- 源文件类型: 只支持 PDF。
- 订单创建方式: 纯后台手动创建订单（不做小红书自动同步，不做 CSV 导入）。
- 邮件发货: 使用 SMTP。

开放问题（不阻塞 MVP，但会影响上线形态）:

- SMTP 服务提供方与账号策略（专用发信账号、授权码、是否要求 DKIM/SPF）。
- 源文件存储位置: 本地存储还是 OSS/COS 私有桶（MVP 可本地，生产建议对象存储）。
